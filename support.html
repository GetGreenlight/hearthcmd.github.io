Support & Setup Guide

Getting Started

For a detailed walkthrough including how the rules system works, see the setup guide.

1. Install the CLI

curl -sSL https://hearthcmd.com/install.sh | bash

Installs the hearth binary to /usr/local/bin. Pre-built binaries are also available from GitHub releases (see that page for macOS Gatekeeper instructions).

2. Log In

hearth login you@example.com

Hearth Cmd emails a 6-digit code to that address. Paste it at the prompt to enroll this machine as a host under your account. First-time users are also asked for a display name and an organization name.

3. Start the Daemon

hearth start

The daemon listens for spawn requests from the app. hearth status shows whether it's running; hearth stop shuts it down. The daemon does not auto-start on boot.

4. Sign In on Your Phone

Download Hearth Cmd from the App Store, enter the same email, and enter the OTP that arrives in your inbox. The host you just enrolled shows up in the host list.

5. Spawn an Agent from the App

Tap + New Agent, pick the host, pick a working directory, pick the agent (Claude Code, Copilot CLI, Cursor CLI, Codex CLI, or Pi), and tap Start. The agent runs on your laptop in yolo mode while approvals and the activity stream go to your phone.

CLI Reference

Command Description
hearth login <email> Log in via email OTP and enroll this host. --invite <token> joins an existing org.
hearth start Start the daemon on this host.
hearth stop Stop the daemon.
hearth status Show daemon status.
hearth talk Open a TUI to talk to active agent instances from this terminal.
hearth org Manage organization, users, hosts, devices, agents, and invites.
hearth wd Manage working directories (list, create, update, abandon, delete).
hearth update Update hearth to the latest version.
hearth logout Sign out on this machine: stops the daemon, revokes credentials, clears config.

Run hearth <command> --help for details. Config lives in ~/.hearth/config.

Using the App

Permission Decisions

Each request can be responded to with four actions:

  • Always Allow — Approves and creates a rule so future matching requests are auto-approved.
  • Allow — Approves this one request.
  • Deny — Denies this one request.
  • Deny & Stop — Denies and sends a halt signal to stop the agent entirely.

How Rules Work

Tapping "Always Allow" generates a pattern that auto-approves future matching commands without interrupting you.

Patterns are subcommand-aware. The system knows that git push and git status are different operations:

  • go build ./cmd/foo → pattern Bash(go build **) — matches any go build
  • git push origin main → pattern Bash(git push **) — matches any git push, but not git reset
  • aws s3 cp file.txt s3://bucket/ → pattern Bash(aws s3 cp **) — three-word prefix for AWS/GCloud

Commands like rm, sudo, kill, and chmod use exact matching. Granting rm -rf /tmp/coverage creates a rule that matches only that exact string — not rm -rf /home.

For compound commands like git add . && git commit -m "msg" && git push, the system splits into segments and generates a separate pattern for each. All segments must have matching rules for auto-approval. Each segment is color-coded by risk (green/orange/red) and segments already covered by rules show a checkmark overlay.

For Read, Write, and Edit tools, patterns match on the directory (e.g., Edit(/home/user/project/src/**)).

Managing Rules

The Grants tab shows all your "Always Allow" rules grouped by project. Swipe left on any rule to delete it.

Talk to Your Agent from Your Phone

The Activity tab shows an input bar at the bottom. Anything you type is sent directly to your agent session as if you typed it at the terminal. Use this to answer prompts, provide additional context, or nudge your agent in a different direction.

Talk to Your Agent requires Hearth Cmd Pro. Pro also includes push notifications so you never miss a request.

Notifications

Hearth Cmd uses Time Sensitive notifications that can break through Focus modes. Make sure notifications are enabled in Settings → Hearth Cmd → Notifications and that Time Sensitive notifications are allowed in your Focus mode settings.

Troubleshooting

Host not appearing in the app

  • Make sure the daemon is running: hearth status.
  • Confirm you're signed in to the same account on the CLI and the app — hearth login emails an OTP to the address you specify, and the app's email must match.
  • Check the About tab in the app — status should show "Connected" with a green dot.

Agent won't spawn from the app

  • Confirm hearth status shows the daemon as running on the target host.
  • Make sure the working directory exists on that host and is readable.
  • Make sure the agent binary you picked (e.g. claude) is installed and on the daemon's PATH.
  • Check the daemon log for errors. Path is shown by hearth status.

Talk to Your Agent not working

  • Talk to Your Agent requires a Hearth Cmd Pro subscription.
  • The agent must have been spawned via the app (or via the daemon on this host).
  • Check that the Activity tab shows an input bar at the bottom when viewing the session.

Not receiving push notifications

  • Check that notifications are enabled: Settings → Hearth Cmd → Notifications.
  • If using a Focus mode, make sure Hearth Cmd is in the allowed apps list or Time Sensitive notifications are enabled.
  • Push notifications are sent when the app is backgrounded. If the app is in the foreground, requests arrive directly via WebSocket.

Requests timing out

  • The default timeout is 590 seconds.
  • If the app is not connected and push notifications fail, the request will time out and be denied.
